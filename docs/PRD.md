## 1. 배경

사내 방화벽 요청 처리는 여러 운영 부서에 분산되어 있다. 임직원이 방화벽 신청을 했을 때, 한 부서가 작업을 완료해도 **다른 부서의 작업이 남아있어 실제 통신이 가능해지기까지는 추가 시간이 더 걸리는 경우가 많다**. 신청자는 자신이 어느 단계에 와 있는지, 다른 부서 작업이 필요한지조차 모른 채 결과를 기다리고, 통신이 안 되는 것을 뒤늦게 발견하고 추가 요청을 하게 된다. 이 과정의 불투명성이 신청자 불만의 주된 원인이다.

## 2. 문제 정의

1. **분산된 처리 주체**: 한 건의 방화벽 요청이 여러 부서를 거치는데, 신청자는 그 흐름을 알 수 없다.
2. **사전 진단 부재**: 신청 전에 "이 통신이 어떤 작업을 필요로 하는지(방화벽 신청? 라우팅?)" 를 알 방법이 없다.
3. **중복·불필요 요청**: 이미 열려 있는 통신에 대해서도 신청이 들어오거나, 라우팅이 필요한 건에 방화벽 신청만 들어와 반려된다.

## 3. 목표 (Goals)

- 신청자가 **Source IP / Destination IP / Port**를 입력하면, 시스템이 **자동으로 어떤 조치가 필요한지 판정**한다.
- 판정 결과를 신청자가 바로 이해할 수 있게 보여주고, 필요 시 **CSR(Customer Service Request)을 한 번에 자동 생성**한다.
- 여러 룰을 한 화면에서 같이 검토할 수 있어야 한다 (`+` 버튼으로 룰 추가).

### 비목표 (Non-Goals, Pilot 단계)

- 실제 사내 CSR 시스템 연동 (Mock 응답으로 대체)
- 실제 CMDB / 네트워크 장비 연동 (Firestore에 가짜 데이터 Seed)
- 사내 SSO / 권한 관리 (Mock 사용자)
- 요청 처리 상태 추적·알림 기능 (본 개발 단계에서 다룸)
- 모바일 최적화

## 4. 사용자

| 페르소나 | 역할 | 사용 시나리오 |
|---|---|---|
| 신청자 (임직원) | 업무 수행 중 통신이 필요한 사용자 | IP/Port 입력 → 결과 확인 → CSR 신청 |
| 운영 담당자 (작성자 본인 부서) | 방화벽 운영 부서 | (파일럿 외) — 본 개발 시 처리 화면 추가 예정 |

파일럿은 **신청자 관점의 입력·진단·신청 흐름**만 다룬다.

## 5. 핵심 사용자 시나리오

```
[1] 사용자가 시스템 접속
[2] Source IP, Destination IP, Port 입력
    └─ 룰을 더 추가하고 싶으면 [+] 버튼으로 행 추가
[3] "조회" 버튼 클릭
[4] 각 룰별로 자동 판정 결과 표시
    - IP에 매칭되는 시스템명 자동 표시
    - 5가지 결과 케이스 중 하나로 라벨링
[5] "방화벽 신청 필요" 또는 "라우팅 작업 필요"가 하나라도 있으면
    [CSR 요청] 버튼 활성화
[6] CSR 폼 작성 (요청자 정보, 사유, 희망일, 기간, 영향도)
[7] [요청하기] → "CSR 생성 완료" Mock 결과 표시
```

## 6. 기능 요구사항

### F1. 룰 입력 (다중)
- 한 행 = 한 룰 (Source IP / Destination IP / Port / Protocol)
- `+` 버튼으로 룰 행 추가, `-` 또는 휴지통 아이콘으로 삭제
- 클라이언트 측 IP 형식 검증 (IPv4)
- Port: 단일 포트(80), 다중 포트(80,443), 범위(8000-8100) 지원
- Protocol: TCP / UDP / ICMP / ANY 선택

### F2. IP → 시스템명 자동 조회
- IP 입력 즉시(debounce 300ms) Firestore `systems` 컬렉션에서 검색
- 매칭되는 시스템명·서버팜·Zone을 룰 행 옆에 표시
- 매칭 없으면 "미등록 IP" 회색 라벨 표시

### F3. 자동 판정 로직 (5케이스)

| # | 케이스 | 판정 결과 | UI 표시 |
|---|---|---|---|
| 1 | Source 또는 Dest IP가 CMDB 미등록 | 자산 등록 선행 필요 | ⚠️ IP 미등록 |
| 2 | 두 IP 사이 네트워크 경로 없음 | 라우팅 작업 필요 | 🔧 라우팅 필요 |
| 3 | 경로 있음 + 룰 허용 | 신청 불필요 | ✅ 통신 가능 |
| 4 | 경로 있음 + 룰 차단/미정의 | 방화벽 신청 필요 | 🔥 방화벽 신청 |
| 5 | Source/Dest가 같은 서버팜 | 신청 불필요 | ✅ 동일 구간 |

판정 알고리즘:
1. Source/Dest IP의 Zone 확인 (CIDR 매칭)
2. 두 Zone 사이를 잇는 방화벽 경로 탐색 (토폴로지 데이터 기반)
3. 경로 상의 모든 방화벽에서 (Src, Dst, Port, Protocol) 룰을 조회
4. 케이스 1~5 중 하나로 분류

### F4. CSR 폼

판정 결과에 "방화벽 신청 필요" 또는 "라우팅 작업 필요"가 하나라도 포함된 경우에만 [CSR 요청] 버튼 활성화.

폼 항목:
- 요청자 이름 (text)
- 사번 (text)
- 부서 (text)
- 요청 사유 (textarea)
- 희망 완료일 (date)
- 사용 기간: 영구 / 임시 (radio) — 임시 선택 시 종료일 추가
- 업무 영향도: 긴급 / 보통 / 낮음 (radio)

### F5. CSR 생성 (Mock)
- [요청하기] 클릭 시 Firestore `csr_requests` 컬렉션에 저장
- 가짜 CSR 번호 발급 (예: `CSR-20260514-0001`)
- "CSR-XXXX 번호로 접수되었습니다" 메시지 + 신청 내용 요약 표시

## 7. 비기능 요구사항

| 항목 | 요구 수준 |
|---|---|
| 응답 속도 | IP 조회 < 500ms, 판정 < 1s (룰 1건 기준) |
| 동시 사용자 | 파일럿 단계 ~10명 (Firestore 무료 티어 내) |
| 브라우저 | Chrome / Edge 최신 버전 |
| 반응형 | 데스크톱 우선 (모바일은 본 개발 단계) |
| 가시성 | 색약 사용자 고려 — 색 + 아이콘 + 텍스트 병행 |

## 8. 데이터 모델 (Firestore)

```
systems (24개)
  └─ { id, name, ip, farm_id, zone_id, description }

server_farms (6개)
  └─ { id, name, cidr, firewall_id }

network_zones (7개)
  └─ { id, name, cidr }

firewalls (5개)
  └─ { id, name, between_zones: [zone_id, zone_id] }

firewall_rules (~11개)
  └─ { id, firewall_id, src_cidr, dst_cidr, ports, protocol, action }

csr_requests (런타임 생성)
  └─ { id, requester, dept, reason, due_date, duration, impact, rules: [...], created_at }
```

## 9. 토폴로지 / Seed 데이터 (파일럿)

- 시스템 24개 / 서버팜 6개 / 방화벽 5개 / Zone 7개 / 룰 11개 / 스위치 100대(개념적, DB 미저장)
- 별도 토폴로지 다이어그램: [`topology.html`](../topology.html)
- 사무실 Zone(10.10.0.0/16)은 CIDR만 정의, 개별 IP는 CMDB 미등록 → 케이스 1 테스트용
